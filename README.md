

# 🍎 NutriCore API

**NutriCore** — это современный бэкенд для веб-приложения по подсчету калорий и управлению диетой. Проект построен на высокопроизводительном фреймворке FastAPI и использует продвинутые механизмы работы с базой данных и безопасностью.

## 🚀 Технологический стек

*   **Язык:** Python 3.12+
*   **Фреймворк:** FastAPI
*   **База данных:** PostgreSQL
*   **ORM:** SQLAlchemy 2.0 (Асинхронный режим)
*   **Миграции:** Alembic
*   **Валидация:** Pydantic v2
*   **Авторизация:** JWT (Access & Refresh Tokens) в защищенных **HttpOnly Cookies**
*   **Менеджер пакетов:** Poetry

## 🛠 Установка и запуск

1.  **Клонируйте репозиторий:**
    ```bash
    git clone https://github.com/yourusername/NutriCore_backend.git
    cd NutriCore_backend
    ```

2.  **Установите зависимости через Poetry:**
    ```bash
    poetry install
    ```

3.  **Настройте переменные окружения:**
    Создайте файл `.env` в корне и добавьте:
    ```env
    DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/nutricore
    SECRET_KEY=ваш_очень_секретный_ключ
    ALGORITHM=HS256
    ```

4.  **Примените миграции базы данных:**
    ```bash
    poetry run alembic upgrade head
    ```

5.  **Заполните базу начальными продуктами (из CSV):**
    ```bash
    poetry run python import_csv.py
    ```

6.  **Запустите сервер:**
    ```bash
    poetry run uvicorn app.main:app --reload
    ```

## 📖 Документация API (Endpoints)

### 🔐 Авторизация (Auth)
Работает на JWT-токенах, которые передаются в `HttpOnly` куках для защиты от XSS-атак.

*   `POST /auth/register` — Регистрация нового пользователя. Принимает рост, вес, возраст, пол и цели.
*   `POST /auth/login` — Вход в систему. Проверяет пароль через `bcrypt` и выдает два токена: `access_token` (30 мин) и `refresh_token` (7 дней).
*   `POST /auth/refresh` — Обновление `access_token` без повторного ввода пароля.
*   `POST /auth/logout` — Выход из системы (удаление кук из браузера).

### 👤 Пользователь (Users)
*   `GET /users/me` — Получение профиля текущего пользователя (включая автоматический расчет дневной нормы калорий).
*   `PUT /users/me` — Частичное обновление данных профиля (вес, рост, цель, уровень активности).

### 🥦 Продукты (Products)
База содержит более 1000+ системных записей. Реализован профессиональный **нечеткий поиск** (Fuzzy Search) на уровне БД.

*   `GET /product/search` — Поиск продукта по названию. Использует расширение `pg_trgm` (триграммы), находит совпадения даже с опечатками (например, «памидор»).
    *   *Параметры:* `name`, `page`, `size`.
    *   *Логика:* Возвращает только системные продукты и личные продукты текущего пользователя.
*   `POST /product/me` — Создание собственного продукта. Он будет виден в поиске только создателю.
*   `DELETE /product/{id}` — Удаление личного продукта (системные продукты удалять запрещено).

### 🍳 Рецепты (Recipes)
Позволяет объединять продукты в сложные блюда.

*   `POST /recipe/` — Создание рецепта. Принимает название и вложенный список ингредиентов (ID продукта + вес).
*   `GET /recipe/` — Список всех созданных пользователем рецептов.
*   `GET /recipe/{id}` — Детальная информация о рецепте со всеми ингредиентами.
*   `DELETE /recipe/{id}` — Удаление рецепта. Поддерживает каскадное удаление (автоматически удаляет записи в дневнике с этим рецептом).

### 📒 Дневник питания (Diary)
*   `POST /diary/` — Добавить еду в дневник. Можно добавить либо `product_id`, либо `recipe_id`.
*   `GET /diary/{date}` — Список съеденного за конкретную дату. Сразу возвращает посчитанные калории и БЖУ для каждой порции.
*   `PUT /diary/{entry_id}` — Изменение веса порции или типа приема пищи.
*   `DELETE /diary/{entry_id}` — Удаление записи.
*   **`GET /diary/summary/{date}`** — Главная аналитическая ручка. Возвращает:
    *   Сумму съеденных КБЖУ за день.
    *   Дневную норму пользователя (цель).
    *   Остаток калорий до цели.
    *   Перебор калорий (если цель превышена).

### ⚖️ Трекер веса (Weight)
*   `POST /weight/` — Записать текущий вес. Если запись на эту дату уже есть — она обновится. Автоматически синхронизирует «текущий вес» в профиле пользователя.
*   `GET /weight/history` — История взвешиваний пользователя, отсортированная по дате (идеально для построения графиков).
*   `DELETE /weight/{id}` — Удаление записи. Автоматически откатывает вес в профиле пользователя к предыдущему значению из истории.

## ⚙️ Особенности реализации

### 🧠 Логика расчетов
Проект использует **Формулу Миффлина-Сан Жеора** для расчета базового метаболизма (BMR), учитывая:
*   Коэффициенты физической активности (от сидячего до интенсивного).
*   Цели пользователя (похудение -15%, набор +15%, поддержание).

### 🛠 Жадная загрузка (Performance)
Для исключения ошибки `MissingGreenlet` в асинхронной среде используется глубокий `joinedload`. При запросе дневника система за один SQL-запрос собирает цепочку:
`Запись -> Рецепт -> Ингредиенты -> Продукты`.

### 🛡 Безопасность
1.  **Ownership Check:** Пользователь никогда не сможет получить, изменить или удалить чужие данные (записи дневника, рецепты, историю веса). Все проверки происходят на уровне БД через `user_id`.
2.  **Validation:** Строгая типизация через Pydantic гарантирует, что в базу не попадут «пустые» даты или отрицательный вес.
